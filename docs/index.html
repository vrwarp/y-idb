y-idb

A persistent, high-performance IndexedDB database provider for Yjs. This library is a robust fork of y-indexeddb, featuring bounded write concurrency, microtask transaction batching, and error propagation.

Key Benefits

  • Bounded Concurrency: The update flusher keeps at most 1 in-flight flush transaction, greatly reducing IndexedDB scheduling locks and wedged connections in WebKit (Safari / iOS browsers). Other operations (set/del, compaction, the page-hide flush, the final flush in destroy()) open their own transactions; a custom transactionRunner can serialize all writes.
  • Microtask Batching: Automatically bundles synchronous or high-frequency updates into a single write transaction with sub-millisecond latency.
  • Durability Safeguards: Best-effort flush listeners queue and persist memory updates on page hide, document destruction, and tab close.
  • Error Handling: Propagates database transaction errors (such as QuotaExceededError) through an Observable event emitter, preventing silent data drops.

Getting Started

Install the library via npm:

npm install --save y-idb

Initialize the persistence provider with your Yjs document:

import * as Y from 'yjs'
import { IndexeddbPersistence } from 'y-idb'

const ydoc = new Y.Doc()
const provider = new IndexeddbPersistence('room-name', ydoc)

provider.on('synced', () => {
  console.log('Database synced! All local updates loaded.')
})

API Reference

new IndexeddbPersistence(docName: string, ydoc: Y.Doc, options?: { writeDebounceMs?: number })

Creates a persistence provider instance. docName acts as the unique IndexedDB database store key. The optional writeDebounceMs sets the aggregation window (defaults to 0, which schedules flushes immediately on the next microtask).

provider.on('synced', (idb: IndexeddbPersistence) => void)

Emitted when database connection is synced and initial history has finished loading.

provider.on('error', (err: Error) => void)

Emitted when a transaction or query fails (e.g. storage is full). Failed updates are preserved in memory and retried with exponential backoff (200ms, 400ms, 800ms, ...). After maxRetries failures (5 by default) the retry-exhausted event is emitted and no further retry is scheduled until the next document update.

provider.destroy(): Promise<void>

Waits for any in-flight write to settle, performs a best-effort flush of all remaining buffered updates, and closes the database connection. Always resolves; a failure of the final write is reported through the error event.

provider.clearData(): Promise<void>

Deletes the entire IndexedDB database, removing all updates and meta properties.
